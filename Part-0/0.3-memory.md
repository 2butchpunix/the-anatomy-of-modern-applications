# Where Does Everything Live?

In the previous chapter, we learned that launching a program causes the operating system to create a process. That process receives everything it needs to begin execution—its own execution environment, scheduling information, permissions, and resources. We also saw that a process may contain one or more threads, each carrying out different pieces of work while sharing the same application state.

As satisfying as that explanation was, it quietly left an important question unanswered.

When we said that a process receives **"its own memory,"** what exactly did we mean?

Open your browser for a moment.

Perhaps you have fifteen tabs open. Spotify is playing music. Your IDE is indexing a project. Slack is running quietly in the background.

Somewhere, every tab remembers the page you're viewing. Your music player remembers which song is currently playing. Your editor remembers every file you've opened, the cursor position in each one, and even the history required to support **Undo**.

None of that information is stored magically inside the application itself.

It has to live somewhere.

That "somewhere" is memory.

---

# What Is Memory?

Imagine asking a friend to multiply two large numbers in their head.

For a few seconds, they can remember the numbers, keep track of intermediate calculations, and eventually tell you the answer. Their ability to perform the calculation depends not only on knowing *how* to multiply, but also on having somewhere to temporarily hold the information they are working with.

Computers work in much the same way.

A CPU is exceptionally good at executing instructions. It can perform billions of operations every second. But executing instructions alone is not enough. Almost every instruction depends on some piece of information. An instruction might need to read two numbers, compare values, store the result of a calculation, or determine which instruction to execute next. None of this is possible unless that information is available somewhere the CPU can access quickly.

That place is **memory**.

Memory is the computer's temporary workspace. It holds the instructions currently being executed, the data those instructions operate on, and the intermediate results produced along the way. As a program runs, information is constantly being created, modified, and discarded, and memory provides the space where all of this activity takes place.

Notice the word **temporary**.

This chapter focuses on **working memory**, not long-term storage.

---

## Memory Is Not Storage

Whenever you save a document, download a movie, or install an application, that information needs to survive even after the computer is switched off. That is the job of storage devices such as SSDs and hard drives. They are designed to preserve information over the long term, continuing to hold your applications, photos, videos, and documents even when the computer has no power.

Working memory serves a very different purpose. It exists to support computation while the computer is actively working. Before a program can begin executing, the operating system first moves the required parts of that program from storage into the computer's working memory.

The CPU almost never works directly with long-term storage. Every application you launch, every image you open, every video you edit, and every webpage you visit must first be brought into working memory before the CPU can meaningfully process it. This working memory is dramatically faster than long-term storage. The reasons behind that speed difference are rooted in hardware design and lie outside the scope of this book, but they are significant enough to shape how modern computers are built.

This distinction is so fundamental that confusing memory with storage is one of the most common misconceptions among new engineers. Although people often use the terms interchangeably in everyday conversation, they solve entirely different problems.

> *Storage preserves information. Memory enables computation.*

This is why applications often take a moment to launch.

The operating system is not merely "opening" them. It first loads the required parts of the program from relatively slow storage into working memory. Only then can the CPU begin executing those instructions.

Without working memory, the CPU would spend most of its time waiting for information to arrive from storage rather than performing useful work.

The computer therefore needs a fast workspace where information can be read, modified, and discarded while computation is in progress.

---

## Memory Through the Eyes of a Process

Let's reconnect this idea with the previous chapter.

When you launched Chrome, the operating system created a process. One of the very first resources that process received was its own memory space. From that moment onward, memory became the process's private workspace—the place where every piece of information required during execution could be created, accessed, modified, and eventually discarded.

This is why the operating system cannot simply execute a program directly from storage.

A program stored on disk is static. Its instructions remain unchanged until someone updates or replaces the program itself. A running process, however, is constantly evolving. Variables change as calculations are performed. Objects are created and destroyed. Functions are called and return values. Files are opened and closed. Network requests arrive with new data that must be processed before responses are generated. Every moment a process is running, its internal state is changing.

All of those changes need somewhere to exist.

Memory provides that workspace.

Seen from this perspective, memory is much more than a temporary storage area. It is the living workspace of a process—a place where the program transforms from a passive collection of instructions into an active, evolving system. Without memory, a process would have instructions to execute but nowhere to maintain the constantly changing state required to execute them.

# Where Does Working Memory Come From?

By now, we understand why memory exists.

A running application needs somewhere to keep the instructions it is executing, the data it is working with, and the intermediate results it produces along the way. Without memory, even the simplest program would be unable to make progress.

That naturally leads to another question.

**Where is this working memory physically located?**

When developers say an application is "using 2 GB of memory," what are they actually referring to? Is memory simply another name for storage? Is it part of the CPU? Or is it a completely different piece of hardware?

To answer those questions, we need to step away from software for a moment and look inside the computer itself.

---

# Meet RAM

If you've ever purchased a laptop or assembled a desktop computer, you've probably seen specifications such as **8 GB RAM**, **16 GB RAM**, or **32 GB RAM**.

Most people know that "more RAM is better," but surprisingly few understand why.

RAM stands for **Random Access Memory**, and it is the physical hardware that provides the computer's working memory. It is here that the operating system loads the information the CPU needs while programs are running.

When you launch Chrome, the operating system does not execute the program directly from storage. Instead, it copies the required parts of the program into RAM. When you open a photograph, the image is first loaded into RAM. When your IDE opens a project, the files you are actively working with are brought into RAM.

The CPU can retrieve information from RAM quickly enough to keep executing instructions without constantly waiting. If every instruction had to be read directly from storage, the processor would spend most of its time idle, waiting for data to arrive.

RAM is therefore not simply another place to store information.

It is the hardware that makes fast computation practical.

---

# Why Not Store Everything in RAM?

At this point, an obvious question arises.

If RAM is so much faster than storage, why not eliminate SSDs altogether and build computers using only RAM?

On the surface, that seems like the perfect solution. Applications would launch almost instantly, files would be available immediately, and almost every operation would become faster.

Unfortunately, RAM has two characteristics that make it unsuitable for long-term storage.

The first is **volatility**.

RAM only retains information while power is supplied. The moment the computer is switched off, everything stored in RAM disappears. This is perfectly acceptable for browser tabs, variables, or the temporary state of a running application, but disastrous for family photos, source code, or financial records that must survive a restart.

The second is **cost**.

Although RAM has become significantly more affordable over the years, it is still considerably more expensive than storage when measured by cost per gigabyte. Filling an entire computer with hundreds of gigabytes of RAM would make it unnecessarily expensive for most users.

Modern computers therefore rely on both technologies because each solves a different problem.

Storage provides persistence. RAM provides speed. Neither replaces the other. Instead, they complement each other.

> *Storage preserves information. RAM provides fast working memory.*

---

# Physical Memory Has a Limit

So far, RAM sounds almost ideal. It is fast, allows the CPU to work efficiently, and serves as the workspace for every running application. Until one simple question arises.

**What happens when the computer runs out of RAM?**

Imagine a laptop with 8 GB of RAM. You begin your day by opening Chrome. A few browser tabs consume some memory. Next, you launch VS Code, Slack, Spotify, Docker Desktop, and perhaps a virtual machine. As the day progresses, each application requests a little more memory until the total memory required exceeds the 8 GB physically available.

At first glance, the solution seems obvious. The operating system could simply refuse to open any more applications.

That would certainly work, but it would also make for a frustrating user experience.

Clearly, modern operating systems do something more sophisticated.

---

# The Illusion of Infinite Memory

Imagine writing an application. 
Should that application need to know whether the user's computer has 8 GB of RAM or 64 GB?
Should it know which regions of RAM are already occupied?
Should it worry about where its variables are physically stored?
That would make software development unnecessarily difficult.

Instead, the operating system presents every process with a comforting illusion.

From the application's perspective, it appears to own a large, continuous block of memory. Variables are created, objects are allocated, and buffers grow without the application ever needing to know where those bytes physically reside.

The addresses used by the application are **virtual addresses**.
Behind the scenes, the operating system, working together with the processor's **Memory Management Unit (MMU)**, translates those virtual addresses into locations within physical RAM.
This abstraction is known as **virtual memory**.

Notice what just happened.

We did not create more RAM. We changed the way applications think about memory.

---

# Why Virtual Memory Matters

One of the greatest strengths of virtual memory is that it allows every process to believe it owns a large, continuous address space. From the application's perspective, allocating memory is straightforward; it never needs to worry about where that memory physically resides or whether another application is already using nearby regions of RAM. That complexity is hidden behind the abstraction provided by the operating system.

This abstraction brings another important benefit: isolation. We learned in the previous chapter that processes should not interfere with one another. Virtual memory helps enforce that guarantee. Two completely different processes can use exactly the same virtual address, yet those addresses may refer to entirely different locations in physical RAM. Each process experiences the illusion of owning its own private memory, while the operating system quietly keeps them separate.

Perhaps the most powerful consequence of this abstraction is flexibility. Because applications never work with physical addresses directly, the operating system is free to move information around, reclaim unused memory, and reorganize physical RAM without requiring the application to know that anything has changed.

Virtual memory solves many of the limitations of physical memory, but it does not create unlimited resources. At some point, even the operating system reaches the limits of what it can manage efficiently.
